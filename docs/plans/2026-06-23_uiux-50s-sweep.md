## Execution order

**Phase 1 — Issue #55 (toast migration)**
1. Backend audit: per-handler verify which already set toast header
2. Template: point hx-target away from status div to `#main` (or omit) for sites that fully migrate; keep target where errors stay inline
3. Remove redundant `<div id="…-status">` for sites that fully migrate
4. For export PDF handlers, add `runtime.BrowserOpenURL` + toast on success path
5. Build + run appshell tests + smoke-test each migrated site

**Phase 2 — Issue #56 (density)**
1. Browse filters: wrap column toggles in `<details>` when 4+ selected
2. Browse: add 25-row option to page-size dropdown
3. Calendar: drop day cells to `min-h-[90px]` below xl
4. `.ghost-link`: add `py-1`
5. Quality radios: collapse to inline

**Phase 3 — Issue #51 (a11y)**
1. Inventory all `<label>` without `for=` in soldier_card, entry_form, layout
2. Generate unique IDs (page-scoped prefix: `soldier-name`, `entry-confederate-home-status`, etc.)

4. Verify uniqueness when both forms render on same page

**Phase 4 — Issue #57 (fold + modal padding)**
1. entry_form: wrap Bio/Notes/Source Records in `<details>` default-closed
2. Three modals: `p-6 sm:p-8` → `p-6 sm:p-7`

## Risks and mitigations

**#55 risks**:
- Memorial JSON preview + Google dry-run still need inline status (buttons embedded). Easy to miss — keep these as exceptions in the migration.
- Export PDF redirect pattern (`runtime.BrowserOpenURL`) needs Windows path handling — verify the existing helper.

**#51 risks**:
- ID collisions when both soldier_card and entry_form render on same page (e.g., edit view). Use page-scoped prefix or template-injected unique IDs.

**#56/#57 risks**:
- Tailwind classes need rebuild after `*.templ` changes (`make css`).
- `details/summary` may not match existing visual style — needs CSS.
