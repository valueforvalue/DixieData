# Rename Soldier to PersonRecord across types, URLs, and schema

The DixieData glossary (`CONTEXT.md`, `GLOSSARY_ALIGNMENT.md`) defines **Person Record** as the canonical umbrella term for what the codebase still calls **Soldier**. Today every URL is `/soldiers/...`, every facade method starts with `Soldier`, every schema column uses `*_soldier_id`, and the umbrella Go type is `models.Soldier` with a `PersonRecord = Soldier` alias only at the viewmodel boundary. A Phase-over-time rename is the only safe path because URLs are user-visible state (bookmarks, browser history, search index), schema columns are persisted in `.ddbak` backups that must keep loading for years, and JSON field names are embedded in exported archives shared between DixieData users. We chose a phased rename with a typed PersonRecord alias, 301 redirect URL aliases, and additive schema migrations over a single-shot rename because the same-schema bad-build risk is unacceptable for a local-first archive that may be running on a machine without network access during a recovery. The rename ships across one release cycle: type alias in v(N+1), facade rename + URL aliases in v(N+2), schema column rename with migration tooling in v(N+3), removal of compatibility shims in v(N+4). Backwards-incompatible JSON keys and column names are never removed until two releases after their replacement lands, and every bump of `CurrentSchemaVersion` triggers a paired migration note in `docs/migrations/` per the existing `bump-version.ps1` policy. The ADR opts for additive schema migrations with a generated compatibility view layer rather than in-place `ALTER TABLE RENAME COLUMN` because backup archives shipped before v(N+3) must still load, and SQLite has no support for `RENAME COLUMN` across versions older than 3.25. Field-level JSON compatibility uses `MarshalJSON` / `UnmarshalJSON` methods on `PersonRecord` and its sub-types to emit both the new and old keys during the migration window, so a v(N+1) reader can still load a v(N-1) shared archive. The merge-review model renames `MergeReviewConflict.LocalSoldier` and `SourceSoldier` to `LocalRecord` and `IncomingRecord` to align with the glossary; the old field names are kept as `MarshalJSON` aliases. The "Soldier" subtype itself (the entry-type value for a servicemember being researched) is preserved unchanged, because the glossary keeps "Soldier" as a valid Person Record subtype and the entry_type column already documents that contract. Research log evidence-type values that use the bare string `"archive"` are renamed to `"local_archive"` to align with the Local / Shared / Backup / Static Archive family, and the form option label changes from "Archive" to "Local Archive" in the same PR. The schema column rename plan is the highest-risk step and is documented as its own ADR section so a future maintainer can review the column-by-column mapping, the backup-archive compatibility test, and the downgrade path before any `ALTER TABLE` runs in production. URL aliases live for one full release cycle and are removed when the next major version ships; analytics on `/soldiers/...` traffic must be reviewed before the removal PR opens. This ADR supersedes the prior deprecation headers in `internal/models/models.go` (the `PersonRecord = Soldier` alias) and the partial facade rename; the existing alias becomes the production type identity rather than a boundary shim.
