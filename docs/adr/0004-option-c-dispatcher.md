# Replace the custom htmx-clone dispatcher with a single Option C contract

DixieData's frontend used a 4,102-line `frontend/app.js` whose core (~115 lines) was a custom `request()` function that mirrored htmx's behaviour — it stripped `hx-*` attributes on DOMContentLoaded, intercepted clicks and submits, fired `fetch()` calls, and dispatched responses. The custom dispatcher read `X-DixieData-Redirect` and called `window.location.assign()` to navigate. Every Go handler in `internal/appshell/`, by contrast, returned `303 See Other` + `Location: /target` + `HX-Redirect: /target` because the prevailing folklore (encoded in `docs/COMMON_BUGS.md §1.9` and `internal/templates/components/conventions.md`) said htmx 2.x with `hx-swap="none"` swallowed 303 responses without the `HX-Redirect` companion header. The two contracts only worked together by accident: `fetch()` follows 303 by default, the response lands as a `200` to `/jobs/{id}`, `request()` notices `response.redirected === true` and falls through to `window.location.assign(response.url)`. The recurring "export options status pages not landing" bug shipped again with every new export handler (commits `70878ac`, `3612dab`, `11f1c01`, `a6f7fa2`, the Google handlers, the printable PDF modal, the bulk-review, orphan-cleanup, duplicate-audit, and image-import handlers) because the legacy recipe said "add HX-Redirect" — but the dispatcher never read HX-Redirect, so the fix addressed the wrong layer. We chose to collapse to a single `200 OK + X-DixieData-Redirect` contract, replace `request()` with a 32-line `dispatchDixieDataForm`, migrate every Go handler via the new `writeExportRedirect` helper, retag 9 templ files from `hx-post`/`hx-put`/`hx-delete`/`hx-confirm` to `action`/`data-action` + `data-dixie-submit` + `data-confirm`, rewrite the conventions doc to describe the new recipe, and add three source-scan regression nets that fail the build on any reintroduction. The static-archive export opts into `enqueueExportOpt{NativeRedirect: true}` to keep the 303 + Location path because it is reached by a plain `<form method="post">` that the browser follows natively. htmx stays loaded for GET-only polling on `/jobs/active` and `/jobs/{id}` — the only legitimate htmx use in the codebase, since polling returns rendered fragments and never needs the redirect machinery.

## Verification

- `go test ./... -short` — 26/26 packages pass.
- `node audit/smoke.mjs` against `dixiedata-web` (the dev server binary, headless Chromium via Playwright) — 46/46 assertions pass. Exercises the dispatcher end-to-end on every share-page button, the print-config modal, the browse filter change handler, the search input, the debug-mode toggle, and the polling fragments.
- `node audit/discover_export_buttons.test.mjs` — 7/7 tests pass.
- Wails desktop (`DixieData.exe` in WebView2): not verified end-to-end in this environment. The agent runs in a headless session without a display; WebView2 needs a window manager. The build produces a valid binary (the launch produced no Go panics or templ compilation errors; only WebView2's renderer failed to attach to a non-existent display). Manual verification on a developer machine is the remaining gap. The handler code is shared 1:1 between `dixiedata-web` and `DixieData.exe`, so the dev-server smoke covers the dispatcher behavior; the only Wails-specific surface is `runtime.OpenFileDialog` calls which aren't on the redirect path.
