# Adopt a single-source-of-truth token vocabulary for the design system

DixieData's design system will be expressed as a fixed set of named tokens (color, radius, shadow, motion) declared in `tailwind.config.js` and consumed by every component template. Hex literals like `#a88a46`, `#8d7440`, `rgba(141,116,64,0.85)`, `1.7rem`, and `0 16px 32px rgba(23,33,43,0.16)` are scattered across `frontend/tailwind.css` and every template that uses raw Tailwind utility classes; refactoring any single component touches dozens of files and risks visual drift. We chose a Tailwind-theme-backed token layer over CSS custom properties and over a per-component CSS module system because the build pipeline is already Tailwind v3 with `make css` rebuilding `frontend/app.css`, the tokens stay typed (Tailwind IntelliSense surfaces missing values at author time), and the same vocabulary feeds the future v2 component primitives in #74 Phase 1 without forcing a second naming convention. Tokens are locked at this point in time so subsequent PRs (#74 Phase 1 button/card/field/pill/toast/empty-state extractions) reference the canonical names instead of inventing new ones per component.
