# v55 — entry_type CHECK constraint + evidence_type rename

- Add `CHECK (entry_type IN ('soldier','wife','widow','linked_person'))` to
  the `soldiers.entry_type` column. SQLite enforces this on every INSERT
  and UPDATE; future direct-SQL writes (and any future migration) cannot
  introduce an invalid entry type.
- Rename `research_log.evidence_type = 'archive'` → `'local_archive'` to
  align with the glossary (an evidence-type value should be unambiguous;
  the bare word "archive" is a category, not a value).
- Add `internal/models/constants.go` exporting `EntryTypeSoldier`,
  `EntryTypeWife`, `EntryTypeWidow`, `EntryTypeLinkedPerson`, and
  `EvidenceTypeLocalArchive` so templates and viewmodels stop scattering
  string literals.

This bump is the data-model prerequisite for the Tier 2 Soldier →
PersonRecord rename (issue #97). After v97 lands, the constants and
constraint will keep the new names stable.

Manual steps: none. Existing archives migrate transparently because the
CHECK constraint is permissive of all currently-valid entry types and the
`archive` → `local_archive` UPDATE rewrites the one column value in place.

Rollback: `bump-version.ps1` with no `-Force` refuses to roll back schema
versions that added a CHECK constraint, since dropping it requires explicit
verification that no rows violate the dropped constraint.
